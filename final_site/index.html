Final Project Report

Gestra: Can a Webcam Replace a Fighting Game Controller?

A laptop-webcam fighting game experiment built on Street-Pyter, MediaPipe Pose, a rule-based baseline, and a small single-user TCN model.

GitHub Repository

We started this project from the game side, not the model side. The idea was simple: sit in front of a laptop, raise an arm, and make a fighting game character punch. Lean to one side, and the character should move. Stay still, and the character should block. No controller, no keyboard, just the webcam.

The hard part is that fighting games need fast and precise input, but human motion is messy. A person who thinks they are sitting still is still moving a little. A webcam also sees only a 2D image, so some motions that feel obvious to a human are not obvious to the system. This project is about that gap between "action classification works offline" and "game control feels usable in real time."

Motivation

This problem is interesting for a few reasons. First, controller-free gaming is still fun to revisit, but most older motion systems depended on special hardware like the Wii Remote or Kinect. A laptop webcam is much more available. Second, this is not only a recognition problem. The output has to drive a real game loop, so timing, stability, and action mapping matter a lot.

There is also an accessibility angle, but with limits. A camera-based interface could help some players who do not want to use a controller. At the same time, it can exclude players with limited upper-body mobility. So we do not want to oversell it as "more accessible" in a general sense. It is better to say it is an alternative interface idea with real tradeoffs.

Prior Work and Background

Motion-controlled games are not new. The Wii made simple body motion into mainstream game input, and Kinect pushed this further with full-body skeleton tracking. Those systems mattered because they showed that body motion can be a real game input method, not just a gimmick. But they also relied on dedicated hardware. Our project asks what happens if we try to do a much smaller version of that idea with only a laptop webcam.

For pose extraction, this project uses MediaPipe Pose / BlazePose through the Pose Landmarker API. It gives 33 landmarks per frame from a single RGB camera, which is enough to try upper-body action control without depth hardware. For sequence modeling, we ended up using a Temporal Convolutional Network (TCN) after first thinking about an LSTM/RNN design. Bai, Kolter, and Koltun (2018) argue that temporal convolutions are a strong alternative to recurrent models for sequence tasks, and that matched this project well because the game loop only needs a short fixed window.

We also tried to learn from a larger public skeleton-action dataset: NTU RGB+D. It is a strong dataset for action recognition, but it came from Kinect-style skeletons, not laptop webcam MediaPipe landmarks. That difference turned out to matter a lot more than raw dataset size. The base game code comes from Street-Pyter, which we modified only enough to let Player 1 receive named actions from the webcam side.

System Overview

The full runtime pipeline is:

Webcam → MediaPipe Pose → 9 upper-body joints → normalization + velocity + acceleration → 20-frame window → rule-based detector or TCN → named action → Street-Pyter game input → character action

This matters because Gestra is a real-time ML system, not only a classifier notebook. The detector has to keep running while the game runs. It also has to map its output into the game's input format in a stable way. That is why smoothing, cooldowns, and action mapping show up in the project almost as much as the model itself.

Data Collection

The final single-user dataset uses 5 classes: idle, lpunch, rpunch, forward, and backward. The recordings are stored as .npz files containing MediaPipe pose landmarks and valid-frame masks. The important point is that this dataset is small and user-specific. It is not a large benchmark dataset.

Features

MediaPipe gives 33 pose landmarks, but we do not use all of them. The final system keeps only 9 upper-body joints: nose, shoulders, elbows, wrists, and hips. This was a practical decision. At first, we thought full-body tracking would be more impressive. It was not practical with a laptop webcam. The lower body was often off-frame, and the extra joints added noise instead of helping.

The feature pipeline is:

  1. Take the 33 MediaPipe landmarks.
  2. Keep only 9 upper-body joints.
  3. Normalize by shoulder center, so the body is centered each frame.
  4. Scale by shoulder width, so body size and camera distance matter less.
  5. Flatten position features to 27 values per frame.
  6. Add velocity and acceleration, giving 81 values per frame.
  7. Stack 20 frames into one input window.

So the final model input is 20 × 81. This part connects directly to the course: data becomes features, features become a sequence input, and that sequence is used for a downstream control task instead of only a static prediction benchmark.

Models

Baseline: Rule-Based Detector

The first playable version did not need any training data. The rule-based detector checks whether a wrist is above shoulder height for punches, and whether shoulder-center position shifts left or right for movement. Otherwise it outputs idle. This baseline mattered because it made the full pipeline testable before the model worked. It also made debugging easier because we could separate game-integration bugs from training bugs.

The problem is that fixed thresholds are brittle. Different sitting heights, camera angles, and body proportions change what "high enough" or "lean enough" means. So the rule-based version was useful, but not reliable enough as the final answer.

Improved Model: Single-User TCN

The final learned model is a small TCN trained on single-user webcam recordings. It takes a 20-frame pose sequence, uses 2 temporal blocks with 64 channels, and has about 93,701 parameters. The output is one of the same 5 action classes used by the game integration code.

The design changed over time. Early on we thought an LSTM/RNN would be the natural choice for sequence data. In practice, the TCN fit the final use case better. The time window is short, the model is simple to run on each window independently, and there is no hidden state to carry through the game loop. That made real-time integration simpler.

Experiments and Results

Baseline vs TCN

The current offline evaluation in results/experiment_log.csv reports 36.73% validation accuracy for the rule-based detector and 100.00% validation accuracy for the TCN on the same validation split. This is a big gap, but it needs an honest interpretation. It does not mean the model generalizes to everyone. It means the model fit a controlled split from one recorded user's webcam data much better than fixed thresholds did.

Training Curve

The baseline run converges quickly. In the saved epoch log, validation accuracy first reaches 100% at epoch 24, then reaches it repeatedly later. This is good for the single-user calibration case, but it is also a warning sign. On a small dataset, fast convergence can also mean overfitting risk. So we do not want to present the curve as "problem solved." It is better to read it as "the model can fit this setup very well, but the setup is narrow."

Confusion Matrix

The confusion matrix for the current single-user TCN is effectively perfect on this controlled split. That looks impressive, but the same caution applies here. A perfect confusion matrix on one user-specific validation split is not the same thing as robust motion control for new users or new camera setups.

Hyperparameter Sweep

The sweep results are almost too clean. All logged TCN training runs in the current CSV reached 100.00% validation accuracy, including: LR-A, LR-B, LR-C, OPT-B, OPT-C, BS-A, BS-C, WIN-A, and WIN-C. What changed more than accuracy was training time and final training accuracy. The main takeaway is that almost every reasonable configuration could fit this small dataset, so data quality mattered more than optimizer tuning.

Live Game-Control Check

Offline accuracy does not automatically mean the game feels good to play. So we also checked the detector inside the actual game loop. For each active gesture, we tried the motion 20 times and counted it as successful if the intended game action appeared within about one second.

How the Design Evolved

This project changed a lot during implementation. The clean final pipeline came after several ideas failed or turned out to be less practical than they looked at first.

Gestra did not start as a fighting game. Our first version was a much smaller two-player hand-gesture game, saved in the earlier Gesture-Game-main repository. That version was useful because it showed that webcam gestures could feel like game input. But it was also too small for the final project. The actions were limited, and there was not enough room to study timing, failure cases, or whether a learned model was really better than simple rules. So we moved the idea into Street-Pyter, where the same gesture problem became a richer control task: punch, move, block, take damage, and fight an AI opponent.

Was AI Needed?

The honest answer is: not at first. AI was not needed for the first playable demo. The rule-based detector was enough to prove that webcam motion could control the game at all. That baseline was useful because it gave the project a working version before the model worked.

AI started to matter when personalization and temporal patterns became important. A trained model can adapt better to one person's movement style than fixed thresholds can, and a 20-frame window can capture motion context that a single frame misses. But even then, a classifier with 100% validation accuracy is not automatically a good game controller. Calibration, smoothing, cooldowns, and the named-action mapping were also necessary.

Course Connection

This project ended up matching the course pipeline pretty directly: data collection, feature design, model choice, train/validation split, loss and accuracy curves, confusion matrix, hyperparameter tuning, and then a downstream task where the prediction actually changes system behavior. That last part made the project more interesting for us, because a model metric that looks good in a notebook can still feel bad in a real interaction loop.

Ethics

Webcam and body-motion data are sensitive. Local processing is safer than cloud processing, and this project keeps pose extraction and inference on the local machine. Even so, recordings should only be made with consent, and users should know what is stored.

Movement-based interfaces can also exclude people. A system trained on one person's body may not work equally for other bodies, and some users may not be able to perform the required gestures comfortably. For that reason, keyboard fallback should remain available instead of being treated as an old mode that no one should use.

Future Work

  • Collect data from 10 to 20 users instead of one user.
  • Run leave-one-user-out evaluation for a real generalization test.
  • Compare TCN against LSTM and a small Transformer.
  • Add a confidence threshold before firing actions.
  • Improve the calibration and quick-record UI.
  • Make the AI opponent more interesting than a simple random behavior loop.
  • Add more gestures only after the base five are stable.
  • Add a visual feedback overlay during play so the user can see what the detector sees.
